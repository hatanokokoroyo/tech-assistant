# 后端设计原则

> 本文档从后端代码中提取的**可执行规范**。新增端点、服务或模型时必须遵循以下规则。

---

## 一、项目结构

```
backend/app/
├── main.py                  # FastAPI 应用工厂 + lifespan
├── core/
│   ├── config.py            # Pydantic Settings（单例）
│   ├── security.py          # JWT + bcrypt 工具函数
│   └── deps.py              # get_current_user 依赖注入
├── db/
│   ├── base.py              # SQLAlchemy DeclarativeBase
│   └── session.py           # AsyncEngine + sessionmaker + get_db
├── models/                  # ORM 模型（每表一个文件）
├── schemas/                 # Pydantic 请求/响应模型（按领域分文件）
├── services/                # 业务逻辑（无状态 async 函数）
├── ai/
│   ├── client.py            # AiClient（OpenAI SDK → DeepSeek）
│   ├── schemas.py           # Tool Calls JSON 定义
│   └── tools.py             # 工具执行实现
├── api/v1/
│   ├── router.py            # 路由聚合（prefix="/api"）
│   └── <domain>.py          # 各领域端点
└── utils/
    └── path_utils.py        # 沙箱路径解析 + 安全校验
```

### 新增功能的文件映射

| 要做什么 | 在哪里创建/修改 |
|----------|----------------|
| 新增数据库表 | `models/<name>.py` + `schemas/<domain>.py` |
| 新增 API 端点 | `api/v1/<domain>.py` + `schemas/<domain>.py` |
| 新增业务逻辑 | `services/<domain>_service.py` |
| 新增 AI 工具 | `ai/tools.py`（函数）+ `ai/schemas.py`（JSON 定义） |
| 修改认证逻辑 | `core/security.py` 或 `core/deps.py` |
| 修改配置 | `core/config.py` + `.env` |

## 二、命名规范

### 文件命名

| 层 | 规则 | 示例 |
|----|------|------|
| Models | `snake_case.py`，每表一个文件，文件名 = 表概念 | `user.py`, `code_repo.py`, `custom_project.py` |
| Schemas | `snake_case.py`，按领域分组 | `auth.py`, `project.py`, `repo.py` |
| Services | `<noun>_service.py` | `project_service.py`, `conversation_service.py` |
| API 端点 | `snake_case.py`，复数名词 | `projects.py`, `conversations.py`, `ssh_keys.py` |

### 类命名

| 场景 | 规则 | 示例 |
|------|------|------|
| ORM 模型 | **PascalCase** 单数 | `User`, `CustomProject`, `CodeRepo` |
| 请求 Schema | **PascalCase** + `Request`/`Create`/`Update` 后缀 | `RegisterRequest`, `ProjectCreate`, `FileUpdateRequest` |
| 响应 Schema | **PascalCase** + `Response`/`Brief` 后缀 | `UserResponse`, `ProjectResponse`, `RepoBrief` |

### 函数命名

| 场景 | 规则 | 示例 |
|------|------|------|
| API 处理函数 | **snake_case** 动词，无前缀 | `register`, `login`, `list_projects`, `create_project` |
| Service 函数 | **snake_case** 动词 | `list_projects()`, `create_project()`, `save_user_message()` |
| 安全工具 | **snake_case** 动词短语 | `hash_password()`, `verify_password()`, `create_access_token()` |
| 依赖注入 | **get_** 前缀 | `get_db`, `get_current_user` |
| 私有辅助 | **_** 前缀 | `_to_response()`, `_fmt()`, `_get_project()` |

### 变量命名

| 场景 | 名称 |
|------|------|
| 数据库会话 | `db` |
| 当前用户 | `user` |
| 查询结果 | `result` |
| Service 导入 | 直接导入模块名（`from app.services import project_service`） |

## 三、响应规范

### 统一 envelope

所有 REST 端点返回：

```python
{"code": 0, "message": "ok", "data": <payload>}
```

- `code` 始终为 `0`（成功），错误通过 HTTP 状态码 + `HTTPException` 处理
- `message` 始终为 `"ok"`（成功）
- `data` 为实际数据，可以是 `None`、dict、Pydantic 模型

```python
# ✅ 正确
return {"code": 0, "message": "ok", "data": {"id": project.id, "name": project.name}}

# ❌ 错误 — 缺少 envelope
return {"id": project.id, "name": project.name}
```

### 分页列表

```python
return {"code": 0, "message": "ok", "data": {"items": items, "total": len(items)}}
```

> **注意**：当前未实现真正的分页（offset/limit），所有记录一次性返回。

### 创建操作

创建操作返回 `status_code=201_CREATED`。

### 日期格式

所有日期在响应中格式化为字符串：`"%Y-%m-%d %H:%M:%S"`（无时区后缀）。

```python
# ✅ 正确
created_at = obj.created_at.strftime("%Y-%m-%d %H:%M:%S")

# ❌ 错误 — 返回 datetime 对象
created_at = obj.created_at
```

### 响应构建方式

两种方式并存：
1. **内联 dict** — 简单响应直接返回 dict 字面量
2. **辅助函数** — 复杂响应用 `_to_response()` 或 `user_to_response()` 封装

## 四、认证与授权

### 认证流程

```
请求 Header: Authorization: Bearer <JWT>
    ↓
HTTPBearer() 提取 credentials
    ↓
decode_access_token() 解码 JWT → user_id
    ↓
db.get(User, user_id) 查询用户
    ↓
检查 user.deleted_at is None
    ↓
返回 User ORM 对象
```

### JWT 结构

```python
payload = {"sub": str(user.id), "exp": utc_timestamp}
# 算法: HS256, 有效期: settings.jwt_expire_hours (默认 24h)
```

### 依赖注入

```python
async def get_current_user(
    credentials = Depends(bearer_scheme),
    db: AsyncSession = Depends(get_db),
) -> User:
```

- 返回完整的 `User` ORM 对象（不是 Schema）
- 检查 `deleted_at is not None` 拒绝已软删除的用户

### 授权模式

- **无角色控制** — 所有认证用户权限相同
- **所有权隔离** — 每个查询按 `user_id` 过滤

```python
# ✅ 正确 — service 层按 user_id 过滤
project = await project_service.get_project(db, project_id, user.id)

# ❌ 错误 — 不检查所有权
project = await db.get(CustomProject, project_id)
```

## 五、数据库规范

### 会话管理

```python
# db/session.py
engine = create_async_engine(settings.database_url, echo=False, pool_size=5, max_overflow=10)
async_session_factory = async_sessionmaker(engine, class_=AsyncSession, expire_on_commit=False)

async def get_db() -> AsyncSession:
    async with async_session_factory() as session:
        try:
            yield session
        finally:
            await session.close()
```

- `expire_on_commit=False` — 避免 commit 后的延迟加载问题
- 每个请求独立会话，请求结束自动关闭

### 提交模式

每个写操作独立提交：

```python
db.add(obj)
await db.commit()
await db.refresh(obj)
return obj
```

- **无跨 service 的 unit-of-work** — 每个 service 函数独立 commit
- `db.refresh(obj)` 确保获取数据库生成的默认值（如 `id`、`created_at`）

### 查询模式

```python
# 单条查询
result = await db.execute(select(Model).where(Model.field == value, Model.deleted_at.is_(None)))
obj = result.scalar_one_or_none()   # 期望 0 或 1 条

# 列表查询
objs = result.scalars().all()        # 期望多条
```

- **始终使用 `select()` 构建查询**（禁止原始 SQL）
- **始终过滤 `deleted_at.is_(None)`**

## 六、软删除规范

所有包含用户数据的表都有 `deleted_at = Column(DateTime)`。

### 删除操作

```python
# 1. 逻辑删除
obj.deleted_at = func.now()

# 2. 级联软删除（手动，非 SQLAlchemy cascade）
for child in children:
    child.deleted_at = now

# 3. 物理文件删除（如适用）
shutil.rmtree(directory)

# 4. 统一提交
await db.commit()
```

### 查询过滤

**所有查询必须包含 `deleted_at.is_(None)` 条件**：

```python
# ✅ 正确
select(Project).where(Project.user_id == user_id, Project.deleted_at.is_(None))

# ❌ 错误 — 会返回已删除的记录
select(Project).where(Project.user_id == user_id)
```

## 七、Service 层规范

### 设计原则

- **无状态模块级 async 函数**（不是类）
- 第一个参数始终为 `db: AsyncSession`
- 返回 **ORM 对象**或 **`list[dict]`**（不返回 Pydantic Schema）

### 两种返回风格

| 场景 | 返回类型 | 示例 |
|------|----------|------|
| 单条操作（创建/获取/更新） | ORM 对象 | `async def create_project(...) -> CustomProject` |
| 列表操作 | `list[dict]` | `async def list_projects(...) -> list[dict]` |

### 错误处理

- Service **不抛出 HTTP 异常**，返回 `None` 表示未找到
- API 层决定 HTTP 状态码和错误消息

```python
# Service 层
async def get_project(db, project_id, user_id):
    # ... 查询
    return result.scalar_one_or_none()  # 可能返回 None

# API 层
project = await project_service.get_project(db, project_id, user.id)
if not project:
    raise HTTPException(status_code=404, detail="项目不存在")
```

### 日期格式化

Service 列表返回中使用 `_fmt()` 格式化日期：

```python
def _fmt(dt) -> str:
    if dt is None:
        return ""
    return dt.strftime("%Y-%m-%d %H:%M:%S")
```

## 八、Pydantic Schema 规范

### 请求 Schema

```python
class ProjectCreate(BaseModel):
    name: str = Field(..., min_length=1, max_length=200, description="项目名称")
    description: str | None = None
```

- 必填字段用 `Field(...)` + `min_length`/`max_length`
- 可选字段用 `str | None = None`（Python 3.10+ union 语法）
- `description` 参数用于 Swagger 文档（中文）

### 响应 Schema

```python
class UserResponse(BaseModel):
    id: int
    username: str
    created_at: str

    model_config = {"from_attributes": True}
```

- `from_attributes = True` — 允许从 ORM 对象填充
- `created_at: str` — 日期**预格式化为字符串**（不是 datetime）

### 松散类型

部分端点接受 `body: dict` 而非 Pydantic 模型（对话、事件、流式端点）。**新增端点应使用 Pydantic 模型**。

## 九、错误处理规范

### 三层错误策略

| 层 | 策略 | 示例 |
|----|------|------|
| **API 层** | 直接抛 `HTTPException` | `raise HTTPException(404, "项目不存在")` |
| **API 层** | 捕获 service 异常转 HTTP | `except FileNotFoundError: raise HTTPException(404, ...)` |
| **AI 工具层** | 返回错误字符串（不抛异常） | `return f"Error: 文件不存在：{path}"` |

### HTTP 状态码使用

| 状态码 | 场景 | 示例 |
|--------|------|------|
| `400` | 请求参数错误 | 消息为空、克隆失败、分支切换失败 |
| `401` | 认证失败 | 用户名密码错误、Token 无效 |
| `403` | 权限不足 | 文件路径越界 |
| `404` | 资源不存在 | 项目/对话/文件/密钥不存在 |
| `409` | 资源冲突 | 用户名已存在 |

### 错误消息语言

所有面向用户的错误消息（`detail=...`）使用**中文**。

```python
# ✅ 正确
raise HTTPException(404, detail="项目不存在")
raise HTTPException(400, detail="消息内容不能为空")

# ❌ 错误
raise HTTPException(404, detail="Project not found")
```

## 十、AI 模块规范

### AiClient 设计

- **每次请求实例化**，携带 `user_id` 和 `custom_project_id` 上下文
- 使用 `AsyncOpenAI` SDK 对接 DeepSeek API
- **流式优先**：所有调用使用 `stream=True`
- **递归工具循环**：执行工具后调用 `_stream_after_tools()` 获取最终回复

### 工具执行规范

- **纯同步函数**（`subprocess` 调用）
- 所有路径通过 `safe_resolve()` 沙箱化，越界抛 `PermissionError`
- 危险命令黑名单拦截
- 输出截断：文件读取 500 行、搜索 100 匹配
- **不抛异常** — 返回 `f"Error: ..."` 字符串供 AI 解读

### SSE 事件格式

```
event: message_start   data: {"conversation_id": <id>}
event: token           data: {"type": "text", "content": "..."}
event: token           data: {"type": "reasoning", "content": "..."}
event: token           data: {"type": "tool_call_progress", "tool_call_id": "...", "tool_name": "..."}
event: token           data: {"type": "tool_result", "tool_call_id": "...", "tool_name": "...", "content": "..."}
event: message_end     data: {"done": true}
```

- `tool_call_progress` **无 `content` 字段**
- `message_end` 为 `{"done": true}`
- SSE 端点**不走 envelope**（直接 `StreamingResponse`）

### 消息持久化

- 用户消息在流式开始**前**保存
- AI 回复 + 工具消息在流式完成后**保存**（在 generator 内部）
- `touch_conversation()` 在流式前后各调用一次

## 十一、路径沙箱规范

### 目录结构

```
/data/tech-assistant/
  └── {user_id}/
        ├── .ssh/id_rsa
        └── {project_id}/
              ├── instructions.md
              ├── doc/log/
              ├── doc/reference-doc/
              └── {repo_name}/
```

### 安全函数

| 函数 | 用途 |
|------|------|
| `sandbox_root(user_id)` | 返回用户工作区根路径 |
| `safe_resolve(base, path)` | 解析路径并检查不越界（`startswith` 校验） |

- **所有文件操作必须经过 `safe_resolve()`**
- 越界路径抛 `PermissionError`
- AI 工具和用户文件 API 共用同一套沙箱函数

## 十三、文件上传规范

### Multipart Form 参数

FastAPI 中 `File()` 和 `Form()` 必须显式标注。未标注的参数会被当作 query parameter，前端通过 FormData 发送的值会被忽略。

```python
# ✅ 正确
async def upload(
    file: UploadFile = File(None),
    description: str | None = Form(None),
):

# ❌ 错误 — description 被当作 query parameter
async def upload(
    file: UploadFile = File(None),
    description: str | None = None,
):
```

### 文件存在性校验

`File(None)` 在 multipart 请求中可能收到空的 `UploadFile` 对象（不是 `None`）。必须检查 `filename`：

```python
# ✅ 正确
if file and file.filename:
    content = (await file.read()).decode("utf-8", errors="replace")

# ❌ 错误 — 空 UploadFile 对象也是 truthy
if file:
    content = (await file.read()).decode("utf-8")
```

### 文本解码容错

用户上传的文件编码不可控。`decode()` 必须使用 `errors="replace"` 或 `errors="ignore"`，否则非 UTF-8 文件会导致 500。

## 十四、用户输入规范化

### 换行符统一

用户粘贴的内容可能包含 `\r\n`（Windows CRLF）。**保存前必须统一为 `\n`（LF）**，否则依赖 LF 的外部命令（如 `ssh-keygen`、`git`）会解析失败。

```python
content = content.replace("\r\n", "\n").replace("\r", "\n")
```

适用场景：SSH 密钥上传、Markdown 内容保存、配置文件写入。

### 内容格式校验

对格式有严格要求的输入（SSH 密钥、JSON、YAML），在保存前做基本格式校验或自动修复（如在 BEGIN/END 标记与数据之间插入换行），而非依赖外部工具的容错能力。

## 十五、数据库字段长度

VARCHAR 长度必须基于**实际最大长度**估算，而非"看起来够用"：

| 字段类型 | 建议长度 | 反面教训 |
|----------|----------|----------|
| 指纹/哈希 | 256 | SSH 指纹含位数+算法+摘要+注释，通常 80-100 字符，VARCHAR(64) 不够 |
| 文件路径 | 500 | 绝对路径可能很长 |
| 用户名/标题 | 200 | 宁长勿短，VARCHAR 存储成本几乎为零 |
| 短描述 | 500 | — |
| 长文本 | `TEXT` | 不限长度 |

> **经验法则**：不确定长度时用 VARCHAR(256) 或 TEXT。VARCHAR 多分配长度不浪费存储（PostgreSQL 只存实际内容）。

## 十六、子进程与外部命令

### 错误不能静默吞掉

调用外部命令（`ssh-keygen`、`git` 等）时，失败必须记录日志：

```python
# ✅ 正确
result = subprocess.run(["ssh-keygen", "-lf", path], capture_output=True, text=True)
if result.returncode != 0:
    print(f"[ssh-keygen -lf] returncode={result.returncode} stderr={result.stderr.strip()}")
    return ""

# ❌ 错误 — 异常被吞掉，无法定位问题
try:
    result = subprocess.run(...)
    if result.returncode == 0:
        return result.stdout.strip()
except Exception:
    pass
return ""
```

### 多步骤降级策略

外部命令可能因输入格式差异而失败。设计多步骤降级：

```python
# SSH 指纹获取：先直接读取，失败则从私钥导出公钥再读
fingerprint = _run_ssh_keygen_fingerprint(key_path)
if not fingerprint:
    pub_content = subprocess.run(["ssh-keygen", "-y", "-f", key_path], ...)
    # 写临时公钥文件 → 再获取指纹 → 清理临时文件
```

### 子进程超时

所有 `subprocess.run` 必须设置 `timeout` 参数，防止外部命令挂起导致请求阻塞。

以下为当前代码中已知的不一致，新代码不应延续：

| 问题 | 说明 | 应改为 |
|------|------|--------|
| `_fmt()` 重复定义 | 9 处相同的日期格式化函数 | 提取到 `utils/format.py` |
| `_func_now()` 重复定义 | `repos.py` 和 `ssh_keys.py` 各一份 | 提取到 `utils/` |
| 部分端点用 `body: dict` | conversations、events、stream 端点 | 改用 Pydantic 模型 |
| 无真正的分页 | list 接口返回全部记录 | 后续添加 offset/limit |
| CORS 全开 | `allow_origins=["*"]` | 生产环境限制域名 |
| 无全局异常处理 | 无 `@app.exception_handler` | 后续添加统一错误处理 |
