# 前端设计原则

> 本文档是从 `doc/record/前端重构设计.md` 中提取的**可执行规范**。
> 新增页面或组件时必须遵循以下规则，不允许随意使用自定义值。

---

## 一、设计 Token

### 色彩（OKLCH）

| Token | 值 | 用途 | 禁止场景 |
|-------|-----|------|----------|
| `--color-background` | `oklch(0.99 0 0)` | 页面底色 | — |
| `--color-foreground` | `oklch(0.15 0 0)` | 正文文本 | **禁止用纯黑 `#000`** |
| `--color-card` | `oklch(1 0 0)` | 卡片/面板背景 | — |
| `--color-muted` | `oklch(0.96 0 0)` | 次要背景、AI 消息气泡 | — |
| `--color-muted-foreground` | `oklch(0.50 0 0)` | 次要文本、占位符 | — |
| `--color-border` | `oklch(0.92 0 0)` | 边框、分割线 | — |
| `--color-primary` | `oklch(0.55 0.18 250)` | 关键按钮、链接、选中态、品牌 | **仅占页面 ≤ 5%** |
| `--color-primary-foreground` | `oklch(0.98 0 0)` | 主色上的文本 | — |
| `--color-destructive` | `oklch(0.55 0.2 25)` | 删除、错误、危险操作 | — |
| `--color-success` | `oklch(0.62 0.17 155)` | 成功状态 | — |
| `--color-warning` | `oklch(0.75 0.15 75)` | 警告、进行中 | — |

> 在 Tailwind 中直接使用语义类名：`bg-background`、`text-foreground`、`bg-primary`、`text-muted-foreground` 等。
> **禁止硬编码十六进制颜色值**（如 `#409eff`），所有颜色必须通过 CSS 变量 / Tailwind 语义类名引用。

### 排版

| 属性 | 规则 |
|------|------|
| 正文字体 | `font-sans`（Inter） |
| 代码字体 | `font-mono`（JetBrains Mono） |
| 正文字号 | `text-base`（16px） |
| 标题最小字号 | `text-lg`（18px），标题 ≥ 正文 × 1.5 |
| 页面标题 | `text-2xl font-bold`（24px Bold） |
| 卡片标题 | `text-lg font-semibold`（18px Semibold） |
| 次要文本 | `text-sm text-muted-foreground`（14px） |
| 辅助标签 | `text-xs text-muted-foreground`（12px） |

### 间距（8px 倍数）

**禁止使用 8 的非整数倍**（如 `p-[6px]`、`gap-[10px]`、`mt-[14px]`）。

| 场景 | Tailwind 类 |
|------|------------|
| 图标与文本间距 | `gap-1`（4px，唯一例外） |
| 元素内部紧凑间距 | `p-2`（8px） |
| 输入框内边距 | `px-3 py-1`（12px / 4px） |
| 组件内边距 | `p-4`（16px） |
| 页面边缘留白 | `p-6`（24px） |
| 区块间距 | `space-y-8`（32px） |

**层级递增关系**：元素间距(8) < 组件内边距(16) < 组件间距(24) < 区块间距(32-48)

### 圆角

| 场景 | 类名 |
|------|------|
| 小元素（Badge、小按钮） | `rounded-md`（8px） |
| 输入框、普通按钮、卡片 | `rounded-md`（8px） |
| 大卡片、对话气泡、模态框 | `rounded-lg`（12px） |
| 头像、圆形按钮 | `rounded-full` |

### 阴影

| 层级 | 类名 | 场景 |
|------|------|------|
| 轻 | `shadow-sm` | 按钮、输入框 |
| 中 | `shadow-md` | 卡片、下拉菜单 |
| 重 | `shadow-lg` | 模态框 |

## 二、组件规则

### 按钮层级

同一视觉区域内 **Primary 按钮最多 1 个**。

| 层级 | variant | 场景 |
|------|---------|------|
| **主操作** | `default` | 提交、发送、创建、保存 |
| **次要操作** | `outline` | 取消、返回 |
| **最次要** | `ghost` | 工具栏图标按钮、Tab 切换 |
| **危险操作** | `destructive` | 删除、清空（必须二次确认） |
| **链接** | `link` | 文本内可操作链接 |

### 表单

- 校验：**Zod schema + React Hook Form**（`zodResolver`）
- 错误显示：输入框下方 `<p className="text-xs text-destructive">` 
- 提交按钮：loading 时显示 `<Loader2 className="h-4 w-4 animate-spin" />` + `disabled`
- 提交成功：`toast.success()`
- 提交失败：`toast.error()`，表单保留用户输入

### 对话框

- 创建/编辑类操作：`<Dialog>`
- 删除等危险操作：`<AlertDialog>` + 确认文案具体说明后果
- 对话框标题：`<DialogTitle>`
- 对话框描述：`<DialogDescription>`

### 空状态

每个列表/内容区域在无数据时显示空状态：

```
图标（h-10 w-10 text-muted-foreground/30）
+ 简短说明文字（text-sm text-muted-foreground）
+ 可选的操作按钮（variant="link"）
```

### 骨架屏

列表加载时使用 `<Skeleton className="h-XX w-full" />`，模拟最终布局高度。

## 三、交互规则

| 场景 | 规则 |
|------|------|
| 可点击元素 | **必须有** `hover:bg-accent` 或等效 hover 状态 |
| focus 态 | `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2` |
| 禁用态 | `disabled:opacity-50 disabled:pointer-events-none` |
| 加载中 | 按钮 Spinner + disabled，页面 Skeleton |
| 危险操作 | AlertDialog 二次确认，文案说明后果 |
| 操作反馈 | Toast 通知（sonner），成功绿色，失败红色 |
| 快捷键 | 文件编辑器 Ctrl+S 保存 |
| 动画时长 | 150-200ms，支持 `prefers-reduced-motion` 禁用 |

## 四、响应式规则

| 断点 | 布局 |
|------|------|
| < 768px | 单列，侧栏用 Sheet 抽屉，字体最小 16px |
| 768px - 1023px | 双列，侧栏折叠为图标栏 |
| ≥ 1024px | 三栏布局（240px + 280px + flex-1） |

- 使用 Flexbox 布局，**禁止 `position: absolute` 做主要结构定位**
- 图片设置 `max-width: 100%`

## 五、代码规则

### 新增 UI 组件

使用 `npx shadcn@latest add <component>` 添加 shadcn/ui 组件到 `src/components/ui/`。
**禁止修改 shadcn/ui 组件的导出接口**，如需扩展在 `src/components/shared/` 中封装。

### 新增页面

1. 在 `src/pages/<section>/` 下创建页面文件
2. 在 `src/router/index.tsx` 中注册路由
3. 数据获取用 TanStack Query（`src/queries/`），不直接在组件中调 `api/`
4. 表单用 React Hook Form + Zod schema

### Tailwind 使用

- 使用语义类名（`bg-background`、`text-foreground`），不使用原始色值
- 条件类名用 `cn()` 工具函数（`clsx` + `tailwind-merge`）
- 间距使用 8px 倍数对应的 Tailwind 类（`p-2`、`p-4`、`p-6`...）
- 字号使用预定义的 text-xs/sm/base/lg/xl/2xl

### 对比度检查

新增自定义颜色组合时，确保：
- 正文文本与背景 ≥ 7:1（WCAG AAA）
- 大标题与背景 ≥ 4.5:1（WCAG AA）
- 按钮文本与按钮背景 ≥ 4.5:1

## 六、API 集成规则

### 6.1 响应 envelope 解包

后端所有 REST 接口返回统一 envelope：`{ code: 0, message: "ok", data: ... }`。

- **api-client 已自动解包 `data` 层** — API 函数（`src/api/`）的返回类型即 `data` 的内容，无需再次取 `.data`
- **分页接口额外解包** — list 接口的 `data` 为 `{ items: T[], total: number }`，必须在 TanStack Query hook 中解包：

```typescript
// ✅ 正确：query hook 中解包 items
export function useProjects() {
  return useQuery({
    queryKey: queryKeys.projects,
    queryFn: async () => {
      const res = await projectApi.list(); // res 类型为 { items, total }
      return res.items;                    // 返回 Project[]
    },
  });
}

// ❌ 错误：直接返回分页对象
queryFn: () => projectApi.list(), // 组件拿到的是 { items, total } 而非数组
```

### 6.2 字段命名对齐

TypeScript 接口字段名**必须与后端响应 JSON 完全一致**（后端使用 snake_case）。

- 新增 API 函数时，**先查 `doc/API定义.md` 或后端代码确认实际字段名**，再定义 TypeScript 类型
- **禁止臆造字段名**（如后端返回 `url` 不能写成 `git_url`）
- 字段名不一致是最难调试的 bug 类型之一——编译通过但运行时拿到 `undefined`

### 6.3 类型必须对齐后端实际响应

`doc/API定义.md` 是初始设计，**后端实际实现可能有偏差**。新增或修改 API 函数时：

1. 先查 `doc/API定义.md` 获取设计意图
2. 再查后端路由代码（`backend/app/api/v1/`）确认实际返回结构
3. 如有偏差，以后端实际代码为准，同时更新 `doc/API定义.md`

## 七、状态与认证规则

### 7.1 Token 读取

**所有需要读取 JWT token 的地方，统一从 Zustand 内存状态读取：**

```typescript
import { useAuthStore } from "@/stores/auth-store";
const token = useAuthStore.getState().token;
```

**禁止从 localStorage 直接读取**。Zustand `persist` 中间件写入 localStorage 是**异步**的——`setToken()` 后内存状态立即更新，但 localStorage 可能还未写入。从 localStorage 读会导致竞态条件（登录成功但后续请求无 token）。

适用场景：
- `lib/api-client.ts` 的 `getToken()`
- `hooks/use-sse.ts` 的 fetch 请求头
- `api/ssh-keys.ts` 的 FormData 上传

### 7.2 FormData 上传

`api-client` 只处理 JSON 请求。FormData 上传（如 SSH 密钥文件）绕过 api-client，必须：

1. 手动注入 `Authorization: Bearer <token>` 头
2. 手动检查 HTTP 状态码
3. 手动解包 `{ code, data }` envelope

```typescript
// ✅ 正确模式
upload: (formData: FormData) => {
  const token = useAuthStore.getState().token;
  return fetch("/api/ssh-keys", {
    method: "POST",
    headers: token ? { Authorization: `Bearer ${token}` } : {},
    body: formData,
  }).then(async (res) => {
    if (!res.ok) throw new Error(`上传失败 (${res.status})`);
    const json = await res.json();
    if (json.code !== 0) throw new Error(json.message || "上传失败");
    return json.data as SshKey;
  });
},
```

### 7.3 用户信息恢复

Zustand `persist` 仅持久化 `token`（通过 `partialize`）。用户信息（`user`）不在 localStorage 中。

应用启动时，`ProtectedRoute` 检测到 token 存在后，应调用 `GET /api/auth/me` 获取用户信息并写入 store。**登录接口已返回 `user` 对象时可直接使用，无需额外调用 `getMe()`**。

## 八、路由规则

### 8.1 多面板布局

本项目采用三栏 IDE 布局（侧栏 + 中栏 + 右栏）。React Router 不支持同一路由层级的多个 `<Outlet />`。

**正确做法**：每个页面组件（FilePanel / ChatPanel / RepoPanel）使用 `PanelLayout` 组件自包含完整三栏：

```typescript
export default function FilePanel() {
  return (
    <PanelLayout
      activeTab="files"
      middleHeader={<Header />}
      middleContent={<FileTree />}
      rightContent={<Editor />}
    />
  );
}
```

**禁止**在 layout 组件中渲染两个 `<Outlet />` 期望分别填充中栏和右栏。

### 8.2 URL 参数校验

`useParams()` 取出的参数类型为 `string | undefined`。转为数字前必须校验：

```typescript
const { projectId } = useParams();
const pid = Number(projectId);

// ✅ 在 query hook 中用 enabled 守卫
enabled: !!projectId && !Number.isNaN(pid)

// ✅ 在组件中提前返回
if (Number.isNaN(pid)) return <Navigate to="/projects" replace />;
```

`Number(undefined)` → `NaN`，`Number("abc")` → `NaN`。NaN 会拼入 API URL（`/api/projects/NaN/files`）导致无意义请求。

### 8.3 导航

- **禁止 `window.location.pathname`** — 不触发 React 重渲染，不感知路由变化
- 使用 `useNavigate()` 编程式导航、`useParams()` 读取路径参数、`useLocation()` 读取当前路径

## 九、SSE 流式规则

### 9.1 事件分割

SSE 协议以 `\n\n` 分隔事件。`data:` 字段可能跨多行（每行一个 `data:` 前缀）。

```typescript
// 按 \n\n 分割事件块
const parts = buffer.split("\n\n");
buffer = parts.pop() || ""; // 最后一个不完整的部分保留

for (const part of parts) {
  let eventType = "";
  let dataLines: string[] = [];

  for (const line of part.split("\n")) {
    if (line.startsWith("event: ")) eventType = line.slice(7).trim();
    else if (line.startsWith("data: ")) dataLines.push(line.slice(6));
  }

  const data = JSON.parse(dataLines.join("\n")); // 多行 data 拼接后解析
  // ...
}
```

### 9.2 事件类型

后端 SSE 端点（`POST /api/conversations/{id}/stream`）发送以下事件：

| event | data 结构 | 说明 |
|-------|----------|------|
| `message_start` | `{ conversation_id }` | 消息开始 |
| `token` | `{ type: "text", content }` | 文本片段 |
| `token` | `{ type: "reasoning", content }` | 思考链片段 |
| `token` | `{ type: "tool_call_progress", tool_call_id, tool_name }` | 工具调用中（**无 content 字段**） |
| `token` | `{ type: "tool_result", tool_call_id, tool_name, content }` | 工具调用结果 |
| `message_end` | `{ done: true }` | 消息结束 |

**注意**：`tool_call_progress` 没有 `content` 字段，前端应按 `tool_name` 展示工具名称。

### 9.3 SSE 端点不走 envelope

`text/event-stream` 响应**不包裹** `{code, message, data}`。前端直接解析原始 SSE 事件流，不能调用 `api-client`。token 从 Zustand 内存状态读取（见 7.1）。
