## 🔴 问题 1：对话 #3、#6 — 用户发消息后 AI 完全无响应

### 现象

两个对话都只有 1 条用户消息，API 轮次 = 0，AI 从未产生任何回复。

- **对话 #3**：用户发送"列出你当前可访问的数据库"
- **对话 #6**：用户发送"尝试使用工具list_datasources查询当前可用数据源"

### 可能原因

1. **SSE 连接在 API 调用前断开** — 用户关闭了页面或网络中断
2. **`_init_mcp()` 连接阻塞** — `streamablehttp_client` 连接 MCP Server 可能长时间超时，在此期间流式响应无法启动

### 代码位置

`backend/app/api/v1/stream.py:76`

```python
# 初始化 MCP 客户端（连接数据库查询服务，失败时降级不阻塞）
await client._init_mcp()  # ← 可能长时间阻塞
```

`backend/app/ai/client.py:458-466`

```python
mcp_url = settings.db_query_mcp_url
self._mcp_transport_ctx = streamablehttp_client(mcp_url)
read, write, _ = await self._mcp_transport_ctx.__aenter__()  # ← 无超时保护
```

### 影响

用户看到"发送了消息但没有任何反应"，体验极差。且用户消息已保存但无 AI 回复，导致对话状态不一致。

## 🔴 问题 2：系统提示词声称有 MCP 工具，但实际不可用

### 现象

系统提示词（`_build_messages`）明确告诉 AI：

> "使用 list_datasources 查看可用的数据库及其类型"  
> "使用 query_mysql / query_redis / query_tdengine 对指定数据源执行只读查询"

但实际上这些工具是 MCP 工具，**只有在 MCP Server 连接成功时才注入到工具列表中**。

### 影响链

```
系统提示词声称有 list_datasources
  → AI 相信自己有该工具 → 尝试调用
  → 返回 {"error": "MCP Server 未连接"}
  → AI 退化为用 run_command 执行 "list_datasources"
  → shell 报错: /bin/sh: 1: list_datasources: not found
  → 对话 #7 中 AI 说："我当前可用的工具集不包含 list_datasources"
  → 但系统提示词仍然声称有
```

### 代码位置

`backend/app/ai/client.py:248-277`（系统提示词）

```python
system_prompt = (
    ...
    "## 数据源查询\n"
    "该项目可能配置了数据库数据源。你可以：\n"
    "- 使用 list_datasources 查看可用的数据库及其类型（MySQL/Redis/TDengine）\n"
    "- 使用 query_mysql / query_redis / query_tdengine 对指定数据源执行只读查询\n"
    ...
)
```

`backend/app/ai/client.py:484-486`（工具列表）

```python
def _get_tools(self) -> list[dict]:
    """返回完整工具列表：本地工具 + MCP 数据库工具。"""
    return TOOLS + self._mcp_tools  # ← MCP 连接失败时 _mcp_tools = []
```

### 根因

系统提示词是**静态的**，不会根据 MCP 连接状态动态调整。`MCP_TOOL_NAMES` 在 `schemas.py` 中定义了这些工具名，但它们只有在 `_init_mcp()` 成功时才被注入。

## 🟡 问题 3：安全过滤器误拦截 — `/dev/` 子串匹配过于宽泛

### 现象（对话 #4 第 153 行）

```
Error: 危险命令被拦截：/dev/
```

### 代码位置

`backend/app/ai/tools.py:24`

```python
dangerous = ["rm -rf /", "sudo", "su ", "mkfs", "dd if=", ":(){ :|:& };:", "/dev/"]
```

### 问题分析

`/dev/` 作为子串匹配，会误拦截包含 `/dev/null` 重定向的合法命令：

```bash
find /path -name "*.java" 2>/dev/null        # ← 被误拦截
ls -la /tmp 2>/dev/null                       # ← 被误拦截
grep -r "pattern" . | head -20 > /dev/null    # ← 被误拦截
```

### 影响

AI 的合法 shell 命令被误拦截，导致任务中断。AI 无法使用常见的 `/dev/null` 重定向来清理输出。

## 🟡 问题 4：AI 工具调用策略混乱 — 对不存在的工具反复尝试

### 现象

| 对话 | 轮次 | AI 行为 | 结果 |
|------|------|---------|------|
| #5 第 3 行 | 1 | 用 `run_command` 执行 `list_datasources` | `/bin/sh: 1: list_datasources: not found` |
| #5 第 4 行 | 1 | 调用 MCP `list_datasources` | `{"error": "MCP Server 未连接"}` |
| #7 第 3 行 | 1 | 用 `run_command` 执行 `list_datasources` | `/bin/sh: 1: list_datasources: not found` |

### 根因

系统提示词让 AI 认为 `list_datasources` 是可用工具，但实际工具列表中没有。AI 尝试了两种方式都失败，却没有从错误中学习（对话 #7 仍然重复同样的尝试）。

### 代码位置

`backend/app/ai/client.py:394-430`（`_execute_tool` 方法）

```python
def _execute_tool(self, name: str, args: dict) -> str:
    if name == "run_command":
        return ai_tools.run_command(...)
    elif name == "read_file":
        return ai_tools.read_file(...)
    # ... 没有 MCP 工具的处理分支
```

注意：MCP 工具的执行是在 `_execute_tool_calls` 中单独处理的，但当 MCP 连接失败时，AI 会尝试用 `run_command` 来执行不存在的命令。

## 🟡 问题 5：对话 #4 — AI 幻觉出不存在的工具类别

### 现象（对话 #4 第 46-51 行）

AI 回复中列出了：

> "数据源查询服务：list_datasources、query_mysql、query_redis、query_tdengine"

并将其归类为"当前可用的 MCP 工具/服务"，但这些工具实际不可用。AI 没有先验证工具是否真的可用就列出来了。

### 根因

AI 根据系统提示词的信息直接生成了工具列表，没有通过 `_get_tools()` 验证实际可用的工具。

## 🟠 问题 6：`_init_mcp()` 超时/阻塞影响首次响应时间

### 现象

当 MCP Server 不可达时，`_init_mcp()` 可能需要较长时间才能超时。

### 代码位置

`backend/app/ai/client.py:448-482`

```python
async def _init_mcp(self):
    ...
    mcp_url = settings.db_query_mcp_url
    self._mcp_transport_ctx = streamablehttp_client(mcp_url)
    read, write, _ = await self._mcp_transport_ctx.__aenter__()  # ← 无超时
    self._mcp_session_ctx = ClientSession(read, write)
    self._mcp_session = await self._mcp_session_ctx.__aenter__()
    await self._mcp_session.initialize()
    ...
```

### 影响

在 MCP Server 不可达时，用户等待时间过长才看到 AI 的第一个 token。

| 优先级 | 问题 | 修复方向 | 预估工作量 |
|--------|------|----------|-----------|
| **P0** | 系统提示词与实际工具不匹配 | 根据 MCP 连接状态动态生成系统提示词中的工具说明 | 中 |
| **P0** | 安全过滤器 `/dev/` 误拦截 | 改为精确匹配独立命令，而非子串；或白名单 `/dev/null` | 小 |

| **P2** | 对话无响应时用户无反馈 | SSE 连接建立后立即发送心跳事件；确保 API 调用前有进度提示 | 中 |
| **P2** | AI 幻觉工具列表 | 在系统提示词中只列出 `_get_tools()` 实际返回的工具 | 小 |

## 🔧 修复方案详述

### P0-1：动态系统提示词

**方案：** 在 `_build_messages` 中接收 MCP 连接状态，动态生成系统提示词。

```python
def _build_messages(self, user_message, history, mcp_available=False):
    base_prompt = "你是一个技术助手..."
    
    if mcp_available:
        base_prompt += "\n\n## 数据源查询\n..."
    else:
        base_prompt += "\n\n## 数据源查询\n当前数据源查询服务不可用，无法直接查询数据库。你可以通过搜索代码仓库中的配置文件来间接获取数据库信息。"
    
    ...
```

**涉及文件：** `backend/app/ai/client.py`

### P0-2：修复安全过滤器

**方案：** 将 `/dev/` 改为精确匹配 `dd if=/dev/`，或添加白名单机制。

```python
dangerous = ["rm -rf /", "sudo", "su ", "mkfs", "dd if=/dev/", ":(){ :|:& };:"]
# 或者：使用正则匹配独立的危险命令，而非子串
```

**涉及文件：** `backend/app/ai/tools.py`

### P1-1：MCP 连接超时

**方案：** 给 `_init_mcp()` 添加 `asyncio.wait_for` 超时保护。

```python
async def _init_mcp(self):
    if self._mcp_initialized:
        return
    try:
        await asyncio.wait_for(self._do_init_mcp(), timeout=5.0)
    except asyncio.TimeoutError:
        logger.warning("[MCP] 连接超时（5秒），数据库查询工具不可用")
        self._mcp_tools = []
    except Exception as e:
        logger.warning("[MCP] MCP Server 不可用: %s", e)
        self._mcp_tools = []
    finally:
        self._mcp_initialized = True
```

**涉及文件：** `backend/app/ai/client.py`

### P1-2：工具失败后的降级提示

**方案：** 在 `_execute_tool_calls` 中，当 MCP 工具返回连接错误时，追加系统消息提示 AI 不要再尝试。

```python
# 在 tool_result 中检测 MCP 连接错误
if "MCP Server 未连接" in result_content:
    messages.append({
        "role": "system",
        "content": "注意：数据库查询工具（list_datasources, query_mysql 等）当前不可用。请不要尝试调用这些工具，改为通过搜索代码配置文件获取信息。"
    })
```

**涉及文件：** `backend/app/ai/client.py`
